# DESIGN_SYSTEM_双域客户沟通平台: 响应式设计规范 (Mobile & Tablet)

## 1. 文档定位

本规范是“双域客户沟通平台”桌面端设计系统（`DESIGN_SYSTEM.md`）的**移动端与平板端补充规范**。

*   **适用范围**：双域客户沟通平台 Main 主站管理后台（通过手机浏览器、iPad/平板浏览器访问的场景）。
*   **不适用范围**：`/sidebar/kf`、`/sidebar/contact` 及任何企业微信侧边栏工具页（侧边栏具有独立的宿主环境约束，请参考其专属规范）。

本规范旨在指导设计师与前端工程师，在不改变现有业务逻辑与底层架构的前提下，将企业级管理系统适配到中小屏幕设备上。**核心底线是：移动端/平板端适配不能破坏“选中的页面必须做全面闭环”的产品原则，绝不允许将复杂页面简单降级为“移动端只读”。**

---

## 2. 设计原则

在 Mobile / Tablet 上适配本系统，核心挑战在于**“信息降维与操作收敛”**，同时保持业务完整性。

1.  **保持企业级专业感**：严格延续现有的“专业、高效、AI赋能、严谨”基调。在移动端依然保持大面积留白与浅灰色背景（`Gray-50/30`），使用高饱和度蓝色（`Blue-600`）作为行动点，拒绝过度消费级化。
2.  **业务闭环不可破**：小屏不等于删减业务。核心字段直接展示，次要字段折叠至详情页或 Bottom Sheet。所有在桌面端能完成的配置与操作，在移动端必须提供相应的交互路径，哪怕路径更长。
3.  **操作触手可及**：管理后台包含大量高频操作（如分配、回复、打标签）。在移动端，行内操作应优先收敛为 `...` 更多操作菜单、Bottom Sheet 或底部悬浮操作栏（Sticky Action Bar）；`Swipe Actions` 仅作为经过充分验证后的可选增强交互。
4.  **符合设备直觉**：桌面端的 Hover 态在移动端不存在，必须转化为明确的点击反馈；桌面端的弹窗（Dialog）在移动端应转化为底部抽屉（Bottom Sheet）或全屏子页面。

---

## 3. 断点规范 (Breakpoints)

采用 Mobile-First 的思维，结合 Tailwind CSS 默认断点，定义以下四个核心视图状态：

| 断点名称 | 屏幕宽度 | Tailwind | 布局策略 |
| :--- | :--- | :--- | :--- |
| **Mobile** | `< 768px` | `base` / `sm` | 单列布局。导航收起为汉堡菜单。表格强制转化为卡片列表。弹窗转为 Bottom Sheet 或全屏页。 |
| **Tablet Portrait (竖屏)** | `768px - 1023px` | `md` | 1-2 列布局。左侧导航折叠为 Icon 模式。表格可保持，但需精简列数或允许横向滚动。 |
| **Tablet Landscape (横屏)** | `1024px - 1279px` | `lg` | 接近桌面端体验。左侧导航可展开。支持 Split View（左列表右详情）。 |
| **Desktop** | `>= 1280px` | `xl` / `2xl` | 完整桌面端布局。全量数据展示，复杂多列组合。 |

---

## 4. 栅格与间距系统

在小屏幕上，屏幕空间的每一像素都极其珍贵，间距系统需要从桌面端的 `p-6/gap-6` 整体“降档”。

*   **Mobile (<768px)**
    *   **容器边距 (Margin)**: `16px` (Tailwind: `mx-4`)
    *   **模块间距 (Gap)**: `16px` (Tailwind: `gap-4`)，紧凑区块使用 `gap-3`
    *   **卡片内边距 (Padding)**: `16px` (Tailwind: `p-4`)，内部小区块使用 `p-3`
    *   **圆角降级**: 桌面端卡片为 `rounded-xl` (12px)，移动端为了节省空间，部分列表卡片可降级为 `rounded-lg` (8px)。
    *   **安全区**: 必须适配 iOS 底部 Home Indicator (`pb-safe`) 和顶部刘海屏 (`pt-safe`)。
*   **Tablet (768px - 1023px)**
    *   **容器边距 (Margin)**: `24px` (Tailwind: `mx-6`)
    *   **模块间距 (Gap)**: `20px` 或 `24px`
    *   **卡片内边距 (Padding)**: `20px` (Tailwind: `p-5`) 或保持 `p-6`

## 5. 响应式导航规范

*   **Mobile 导航**：
    *   **顶部栏 (Top Bar)**：固定在顶部，包含汉堡菜单图标（左侧）、页面标题（居中）、全局搜索/消息（右侧）。
    *   **主导航 (Drawer)**：点击汉堡菜单，从左侧滑出全屏宽度的 Drawer 展示完整菜单树，使用 `shadow-2xl`。
    *   **面包屑 (Breadcrumbs)**：在 Mobile 上**隐藏**。替换为顶部栏左侧的“返回”按钮（`< 返回`）。
*   **Tablet 导航**：
    *   **竖屏**：左侧导航栏默认收起为 `64px` 宽度的 Icon 模式。Hover 或点击展开悬浮面板。
    *   **横屏**：与桌面端一致，固定展示 `240px` 宽度的左侧导航。
*   **页面级操作 (Page Actions)**：
    *   桌面端右上角的“新建”、“导出”等按钮，在 Mobile 上应下沉至屏幕底部的 **Sticky Action Bar**，或合并至右上角的 `...` (More) 菜单中。

---

## 6. 页面类型适配规范

1.  **列表页 (List Pages)**
    *   Mobile: 表格转换为 Card List。提供顶部搜索框和筛选按钮（点击唤起 Bottom Sheet 筛选）。
    *   Tablet: 保留表格，但隐藏次要列，或固定首尾列允许中间横向滚动。
2.  **列表 + 详情页 (Split View)**
    *   Mobile: 降级为两个独立页面（Push 路由切换）。点击列表项进入详情页，详情页顶部提供返回按钮。
    *   Tablet: 保持左右分栏（如 35% 列表 / 65% 详情）。
3.  **配置页 (Settings / Forms)**
    *   Mobile: 多列卡片布局改为单列垂直堆叠。复杂的 Tab 切换可改为垂直的 Accordion（折叠面板）或分步表单。
4.  **看板页 (Dashboards)**
    *   Mobile: 统计卡片从横向排列改为 `1x2` 或 `1x1` 的网格。复杂图表允许横向滑动查看细节。

## 7. 关键页面的响应式策略

### 7.1 CustomerList (客户列表)
*   **Mobile 布局策略**: 表格退化为客户卡片列表。
*   **信息保留与折叠**: 卡片头部展示头像+姓名+企微标识（使用现有的 `text-[10px]` 业务标签），中间展示核心标签，底部展示最近跟进时间。隐藏“创建时间”、“来源”等次要列，这些信息折叠进点击后的详情页中。
*   **操作与交互**: 行内操作（编辑、分配）优先收敛到卡片右上角的 `...` 菜单或底部 Sticky Action Bar。`Swipe Actions` 只作为可选增强方案，在宿主环境和交互验证充分时再启用。批量操作通过点击顶部“选择”进入多选模式，底部弹出 Sticky Action Bar（分配、打标签）。
*   **搜索/筛选**: 顶部保留搜索框，复杂筛选收敛为一个“筛选”Icon，点击唤起 Bottom Sheet。
*   **Tablet 布局策略**: 保留表格结构，但隐藏低优先级列，确保核心列（姓名、状态、跟进时间）的阅读体验。

### 7.2 Customer360 (客户 360 视图)
*   **Mobile 布局策略**: 单列瀑布流。
*   **信息保留与折叠**: 顶部固定展示客户基础信息卡片（头像、姓名、电话）。下方使用横向滑动的 Tab 切换视图（跟进记录、订单、业务标签）。
*   **操作与交互**: “写跟进”作为核心业务动作，必须固定在底部悬浮栏 (Sticky Bottom Bar)，确保用户在浏览任何 Tab 时都能随时记录。
*   **Tablet 布局策略**: 采用 Split View。左侧 30% 展示客户档案与标签，右侧 70% 展示跟进时间线（沿用 `border-l-2 border-gray-200` 的时间线样式）。

### 7.3 CSCommandCenter (客服工作台)
*   **Mobile 布局策略**: 列表与详情分离的 Push 模式。默认进入“会话列表”。点击某个会话后，**全屏进入聊天界面**（隐藏左侧列表）。
*   **信息保留与折叠**: 聊天界面的辅助资料信息在手机上默认隐藏，收敛至聊天界面右上角的“资料”入口中，点击后以 Drawer 或 Bottom Sheet 形式滑出。
*   **操作与交互**: 底部输入框固定。快捷回复、转交等操作放入输入框旁边的 `+` 号菜单或顶部栏。
*   **Tablet 布局策略**: 以“会话列表 + 聊天窗口”双栏作为基线。辅助资料区域按剩余空间渐进展开：空间充足时可作为附加面板出现，空间紧张时保持折叠，不将三栏布局写作默认完成态。

### 7.4 TaskCenter (任务中心)
*   **Mobile 布局策略**: 任务卡片列表。
*   **信息保留与折叠**: 顶部使用 Tab 切换任务状态（待办、已办）。卡片展示任务标题、截止时间（带 Clock 图标）、优先级 Badge。任务详情点击后进入全屏子页面。
*   **操作与交互**: 状态流转是核心。不支持复杂的看板拖拽，改为明确的动作按钮：左滑卡片出现“完成”，右滑出现“延期”；或在卡片底部直接提供平铺的“标记完成”小按钮。
*   **Tablet 布局策略**: 采用多列网格布局展示任务卡片，或保持列表布局但增加展示字段（如任务描述摘要）。

### 7.5 ReceptionChannels (接待渠道配置)
*   **Mobile 布局策略**: 渠道卡片单列排布。
*   **信息保留与折叠**: 列表页展示渠道名称、状态（启用/停用）、接入平台。
*   **操作与交互**: 必须保持完整的配置闭环。点击“配置”进入全屏新页面，表单字段垂直堆叠。长表单使用 Accordion 分组，保存按钮固定在底部。
*   **Tablet 布局策略**: 渠道卡片 2 列或 3 列网格排布。

### 7.6 RoutingRules (路由规则配置)
*   **Mobile 布局策略**: 规则列表单列排布 + 分步向导式编辑。
*   **业务闭环要求**: **绝不允许在移动端降级为“仅查看和启停”**。必须提供完整的规则编辑能力。
*   **操作与交互**: 复杂的“If-Then”规则树在手机上无法同屏展示。解决方案是**拆解编辑流程**：
    *   点击规则进入详情页，条件（If）和动作（Then）垂直堆叠展示。
    *   点击“添加条件”，唤起 Bottom Sheet 选择条件类型。
    *   选中类型后，Push 到新页面专门配置该条件的具体参数，配置完返回。
    *   通过“逐级下钻”代替“同屏平铺”，实现移动端的复杂规则编辑闭环。
*   **Tablet 布局策略**: 规则树垂直展开，条件块之间增加明显的连线和触控间距，允许在当前页弹出 Dialog 进行局部编辑。

### 7.7 OrganizationSettings (企业微信集成检查 / 平台组织与权限管理)
*   **Mobile 布局策略**: 页面拆分为两类主区块：上半部分为“企业微信集成检查”状态卡片与能力矩阵，下半部分为“平台组织与权限管理”列表。
*   **信息保留与折叠**: 集成检查区优先展示授权状态、能力状态、缺失项与修复建议；平台权限区展示角色、成员绑定与调试开关等管理项。避免将页面继续定义为“组织树设置页”或“底层技术参数配置页”。
*   **操作与交互**: 集成检查相关内容以只读检查和触发检查命令为主；平台组织与权限管理中的角色、成员、开关配置在 mobile 上进入独立单列配置页或 Bottom Sheet 完成编辑。
*   **Tablet 布局策略**: 以“集成检查概览 + 平台权限管理列表/详情”双区块布局为基线。左侧优先承载检查摘要或角色列表，右侧承载详情与编辑面板，不预设组织树为页面核心结构。

### 7.8 Dashboard (数据看板)
*   **Mobile 布局策略**: 单列或双列流式布局。
*   **信息保留与折叠**: 核心指标卡（KPI Cards）采用 `2列` 网格。数据表格仅展示 Top 5，提供“查看全部”按钮跳转新页面。
*   **操作与交互**: 折线图/柱状图高度固定为 `250px`，隐藏复杂的图例，Tooltip 改为点击数据点触发（而非 Hover）。对于时间跨度长的趋势图，允许图表容器横向滑动（`overflow-x-auto`）。
*   **Tablet 布局策略**: 统计卡片 3-4 列网格，图表可 2 列并排展示。

## 8. 组件级响应式规范

| 组件 | Desktop / Tablet Landscape | Mobile / Tablet Portrait |
| :--- | :--- | :--- |
| **Header (顶部栏)** | 包含面包屑、搜索框、用户信息 | 仅保留标题、汉堡菜单、搜索Icon |
| **Table (表格)** | 完整多列展示，支持复杂排序筛选 | 转化为 Card List。极简表格允许横向滚动 |
| **Dialog (弹窗)** | 屏幕居中，带遮罩 | 转化为 **Bottom Sheet**（从底部滑出），或全屏页 |
| **Filter Bar (筛选)** | 横向平铺在列表上方 | 收敛为一个“筛选”按钮，点击唤起 Bottom Sheet |
| **Search Input** | 展开的输入框 | 默认仅展示 Icon，点击后展开覆盖顶部栏 |
| **Action Button** | 页面右上角或表格上方 | 页面底部 Sticky 悬浮、卡片内主操作按钮或 `...` 更多操作菜单；`Swipe Actions` 仅作可选增强 |
| **Tabs (标签页)** | 横向排列，超出不滚动 | 允许横向滑动（隐藏滚动条），当前项居中对齐 |
| **Form (表单)** | 支持多列布局 (Grid) | 强制单列布局 (100% 宽度)，标签在输入框上方 |

*   **最小触控尺寸**：所有可点击元素（按钮、链接、Icon）的物理触控区域不得小于 `44px * 44px`。

## 9. 数据密度规范

在 Mobile 上，必须执行**“裁剪与折叠”**策略：

1.  **首屏只展示最重要信息**：例如客户列表，桌面端有 8 列数据。移动端卡片只保留：姓名、企微状态、最近跟进时间、负责人。
2.  **次要信息次级展开**：将“创建时间”、“客户来源”、“所在行业”等字段，放入客户详情页的“详细信息”折叠面板（Accordion）中。
3.  **长文本截断**：备注、最新跟进记录等长文本，在列表卡片中最多显示 2 行（`line-clamp-2`），点击进入详情查看完整内容。

---

## 10. 表格与详情的转换规范

**管理后台表格在 Mobile 上的标准转化范式（Card List）：**

*   **主字段 (Primary)**：作为卡片标题，字号加大，加粗（如：客户姓名）。
*   **次字段 (Secondary)**：在标题下方，灰色字体（如：手机号、公司名称）。
*   **元信息 (Meta/Tags)**：使用小尺寸 Badge 横向排列（如：标签、状态）。
*   **关键指标/状态 (Right-aligned)**：卡片右上角展示核心状态（如：高意向、已流失）。
*   **行内操作 (Actions)**：
    *   主推荐：卡片右下角或右上角放置 `...` 按钮，点击呼出 Action Sheet / Bottom Sheet。
    *   可选增强：在宿主环境允许且验证充分时，可增加 Swipe Actions 作为快捷入口。

---

## 11. 表单与配置页规范

1.  **分组与折叠**：超过 10 个字段的表单，在 Mobile 上必须分组。使用 Section Header 隔开，或使用 Accordion 默认收起非必填组。
2.  **保存按钮固定**：长表单的“保存/提交”按钮，必须使用 Sticky 定位固定在屏幕底部（包含安全区 padding），确保用户随时可提交。
3.  **危险操作 (Danger Zone)**：如“删除客户”、“解散部门”，必须放置在页面最底部，使用红色按钮，且点击后必须弹出二次确认（Bottom Sheet 或 Alert）。

---

## 12. 图表与数据看板规范

1.  **简化图表交互**：移动端图表（Echarts/Recharts）必须关闭 Hover 触发的 Tooltip，改为 `Click/Touch` 触发，并在图表上方或下方固定位置展示选中点的数据。
2.  **横向滑动**：对于时间跨度较长的趋势图（如 30 天趋势），不要在小屏上强行挤压，应保持数据点间距，允许图表容器横向滑动（`overflow-x-auto`）。

---

## 13. 状态反馈与空态规范

*   **Loading**：放弃全屏的 Spinner 转圈。全面采用 **骨架屏 (Skeleton)**，保持页面布局在加载前后的稳定性。
*   **Empty (空状态)**：Mobile 上的空状态插画尺寸不得超过 `120px`。必须提供明确的引导按钮（如“去创建”）。
*   **Error (错误态)**：表单校验错误必须在输入框下方原地提示（红色文字），不要使用全局 Toast 提示表单错误。
*   **Offline / Stale**：网络不佳时，在顶部栏下方显示一条细长的黄色警告条（“当前网络不可用，显示缓存数据”）。

---

## 14. 交互规范

*   **手势优先级**：
    *   横向滑动（Swipe）：主要用于 Tab 切换、图表滚动等天然滑动场景。卡片级 Swipe Actions 仅作为可选增强，不作为默认主交互。
    *   边缘右滑（Edge Swipe）：用于返回上一级页面（iOS 原生行为，前端需兼容路由）。
*   **Bottom Sheet vs Drawer**：
    *   **Bottom Sheet**：用于上下文相关的操作（如：筛选、单项操作菜单、选择器）。
    *   **Drawer (左侧)**：仅用于全局主导航。
    *   **Drawer (右侧)**：在 Mobile 上尽量避免，统一改为新页面 Push 或 Bottom Sheet。
*   **Modal 使用边界**：Mobile 上**严禁使用屏幕居中的复杂 Modal**。小提示用 Alert，复杂表单用全屏页，中等内容用 Bottom Sheet。

## 15. 响应式实现建议（给前端工程师）

以下为前端工程落地的**推荐实践**，非强制约束，团队可根据实际技术栈灵活调整：

1.  **CSS 优先的布局控制**：推荐采用 Mobile-First 编写 Tailwind 类名（如 `w-full md:w-1/2 lg:w-1/3`），尽量通过 CSS 媒体查询解决布局变化，减少 JS 层的重绘。
2.  **组件拆分策略**：当 Desktop 表格和 Mobile 卡片差异过大，导致同一个组件内出现大量条件判断时，建议拆分组件。
    *   例如：维护 `<CustomerTable />` 和 `<CustomerCardList />` 两个视图组件。
    *   在父容器中根据视口宽度决定渲染哪一个（可通过 CSS `hidden md:block` 或可选的 `useMediaQuery` Hook 控制）。
3.  **断点命名对齐**：建议统一使用 Tailwind 默认断点（`sm`, `md`, `lg`, `xl`），避免在业务代码中硬编码 `window.innerWidth < 768`。
4.  **安全区适配**：建议在全局 CSS 中配置环境变量，以完美适配移动端异形屏：
    ```css
    .pb-safe { padding-bottom: env(safe-area-inset-bottom); }
    .pt-safe { padding-top: env(safe-area-inset-top); }
    ```

## 16. 设计反模式 (Anti-Patterns)

**在双域客户沟通平台的响应式设计中，绝对不要做以下事情：**

❌ **将复杂页面降级为“只读”**：以“手机屏幕太小”为由，剥夺用户在移动端配置路由规则或修改权限的能力。
❌ **直接缩小桌面表格**：让用户在手机上双指放大看表格是极其糟糕的体验，必须转化为卡片或允许横向滚动。
❌ **引入当前项目不存在的复杂交互**：例如在移动端强行实现看板拖拽，或预设未经确认的复杂三栏联动。
❌ **用过多弹窗堆叠复杂流程**：Mobile 上弹窗叠弹窗会导致用户迷失，应改为分步全屏页。
❌ **在小屏上保留过多同级按钮**：不要把“编辑、分配、打标签、删除”四个按钮并排放在手机卡片上，必须收拢。
